About Spillover Lab

Inspiration for this Lab

This project is a reimplementation of my master's research on GARCH volatility spillover effects in global markets. It examined how a Chinese real estate market collapse would impact global real estate ETFs. Think 2008 Global Financial Crisis but with the epicenter switched to Beijing.

With this tool, the epicenters can be easily switched. Where do you see a bubble brewing?

The original .pdf research is here: GitHub

Features

Data Sources

Access real-time market data from Stooq and Yahoo Finance, or generate synthetic R&D datasets.

  • Stooq and Yahoo Finance: Real-time stock, ETF, and index data
  • Synthetic Data: Controlled testing environments

Econometric Models

Industry-standard time series models for financial analysis and forecasting. Implementation details available at ReadTheDocs.

  • ARIMA Models: Price forecasting and trend analysis
  • GARCH Models: Volatility clustering
  • Spillover Analysis: Interconnection measurement

Visualizations

Charts and analysis tools for comprehensive market research.

  • Time Series Plots: Price and volatility
  • Spillover Networks: interconnection diagrams
  • Risk Metrics: Various point estimates

Project Overview

Spillover Lab is a comprehensive suite of tools for time series analysis and forecasting, focusing on financial and econometric modeling. The project consists of four main components that work together to provide a complete solution.

Architecture

The components interact according to the following architecture:

sequenceDiagram actor User participant Frontend as Frontend participant API as API participant Package as Compute Package participant DB as PostgreSQL DB User->>Frontend: Submit analysis request Frontend->>API: Forward request parameters API->>Package: Execute computations Package-->>API: Return computation results API->>DB: Write results to database API-->>Frontend: Return processed results Frontend-->>User: Display visualization %% Direct API interaction User-->>API: Direct REST/GraphQL request API-->>User: Direct API response

This separation of concerns allows for:

  • Scalable architecture with specialized components
  • Independent development and deployment of each component
  • Reuse of the various components

Tech Stack

Frontend

A Django web app provides an intuitive UI for analysis. Why Django?

  • less boilerplate
  • faster prototyping
  • batteries-included (auth, forms, routing)
GitHub Repository

API

A FastAPI service handles the computational tasks. Why FastAPI?

  • Fast, easy to write
  • auto-docs with Swagger/OpenAPI
  • GraphQL wrapper for flexible data querying

Database

A Postgresql database stores pipeline run metadata and model results. Why postgres?

  • scales
  • good Python libraries (psycopg2, SQLAlchemy)

Compute Package

Python package implements statistical models. Why package?

  • package once, use everywhere (API, CLI)
  • Easy to publish and integrate with CI/CD
